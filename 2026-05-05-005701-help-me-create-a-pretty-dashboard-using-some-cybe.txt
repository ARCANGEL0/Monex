● Honest answer: it works, but it's not a natural fit for this app. Let me lay
  out the options:

  SQLite ⭐ recommended
  - Single file (db.sqlite3), zero config, ships with Python.
  - Perfect for personal/single-user apps — millions of rows handle fine.
  - Backups = copy one file. Works seamlessly with Django ORM.

  PostgreSQL
  - Production-grade SQL DB. Needs a server running (Docker/native).
  - Overkill for a single-user expense tracker, but rock solid if you want it.

  MongoDB (via the official django-mongodb-backend, GA in 2025)
  - Works with Django, but expense data is deeply relational: Transactions →
  Bank, Category; Budgets per category per month; Recurring rules; aggregations
  across joins. That's exactly what SQL is built for.
  - You'd be fighting the data model. JSONField in SQLite/Postgres covers any
  flex-schema needs you might have.
  - Adds: a Mongo server to run, a less-mature Django backend, no real upside
  for this app.

  My strong recommendation: SQLite. Zero ops, blazing fast for this scale,
  perfect for personal use. You said "Django and Python" — SQLite is the
  canonical pairing.

  If you have a reason to want Mongo (already running it, learning it, etc.),
  say so and we'll do it — but for an expense tracker I'd push back. Want me to
  go with SQLite?

● Good context — that changes the calculus. Let me be straight about what
  Cloudflare can/can't do with Django:

  Cloudflare Workers can't run Django. Workers are JS/TS (Python is limited
  Pyodide, no Django). So the "deploy on Cloudflare" path realistically means
  one of:

  1. Cloudflare Containers (GA 2025) — runs your Django container at the edge,
  with persistent volumes. SQLite works but the volume is single-region and tied
   to one container. Restarts/redeploys risk data unless you persist properly.
  2. Cloudflare DNS in front of another host (Fly.io / Railway / Render / a VPS)
   — Django runs there with whatever DB, Cloudflare just proxies + adds
  CDN/auth. xflow.arcangelo.net works the same way.

  Database options for "accessible anywhere":

  - SQLite — works if you have one server with persistent storage. Single-user
  app, low volume → totally fine. But you can't have two regions reading the
  same DB.
  - PostgreSQL on Neon / Supabase free tier — separate managed DB, accessible
  from any host (CF Containers, Fly, etc.). Works perfectly. Looks more
  "production" in your portfolio.
  - Cloudflare D1 (their edge SQLite) — Django can't natively connect (no
  driver). Skip.

  My recommendation: build with DATABASE_URL-based config from day one.
  - Use dj-database-url. Default to SQLite locally (sqlite:///db.sqlite3) — zero
   friction now.
  - When you deploy, set DATABASE_URL=postgres://… (Neon free tier, 5 sec setup)
   and Django switches with no code changes.

  Best of both: SQLite for portfolio repo + local dev (looks clean, no setup
  needed to run), Postgres for production when you flip the env var. Zero code
  rewrite, zero migration headache (Django generates Postgres-compatible
  migrations automatically).

   clean Django, your private repo just adds cf-deploy.sh + a cloudflared
